¿Qué es la Integración Continua?

🔄

Definición simple

La Integración Continua (CI) es una práctica de desarrollo donde cada vez que un desarrollador sube código al repositorio, se ejecutan automáticamente una serie de verificaciones: compilación, pruebas, análisis.

🎯

¿Para qué sirve?

Detectar errores en minutos, no en semanas. Cuando varios desarrolladores trabajan en el mismo proyecto, CI asegura que el código de todos funcione junto antes de integrarlo a la rama principal.

  • • Bugs descubiertos tarde y costosos
  • • "En mi máquina funciona..."
  • • Miedo a hacer cambios grandes
✅

Con CI

  • • Integración continua, poco a poco
  • • Bugs detectados en minutos
  • • Entorno limpio y consistente
  • • Confianza para refactorizar
02

Anatomía del archivo .yml

El archivo .yml vive en .github/workflows/ dentro de tu repositorio. GitHub lo detecta automáticamente. Todo archivo de CI tiene 4 secciones fundamentales:

Define el nombre que aparecerá en la pestaña Actions de GitHub. Es solo descriptivo, no afecta el comportamiento.

YAML
name: CI - Mi Proyecto

Define el evento que dispara el pipeline. Puede ser un push, un pull request, un horario, o incluso un botón manual.

YAML
on:
  push:
    branches: [ main, development ]
  pull_request:
    branches: [ main ]

Variables accesibles desde cualquier job del pipeline. Si necesitas cambiar un valor, lo cambias en un solo lugar.

YAML
env:
  NODE_VERSION: '18'
  APP_NAME: 'mi-app'
💡 Para usar la variable: ${{ env.NODE_VERSION }}

El corazón del archivo. Aquí defines qué hace el pipeline. Cada job corre en su propia máquina virtual limpia de GitHub. Por defecto, los jobs corren en paralelo.

YAML
jobs:
  mi-job:                    # nombre del job
    runs-on: ubuntu-latest  # servidor gratuito de GitHub
    steps:
      - name: Primer paso
        run: echo 'Hola mundo'

Push — Al subir código

Se activa cada vez que alguien hace git push. Puedes limitarlo a ramas específicas o usar comodines con *.

YAML
on:
  push:
    branches: [ main, development, feature/* ]
    # feature/* cubre: feature/login, feature/pagos, etc.

Pull Request — Antes de mergear

Se activa cuando se abre o actualiza un Pull Request. Ideal para validar código antes de que entre a la rama principal.

YAML
on:
  pull_request:
    branches: [ main, development ]
💡 Combina push + pull_request para máxima cobertura.

Manual — workflow_dispatch

Agrega un botón "Run workflow" en la pestaña Actions. Útil para ejecutar el pipeline cuando tú quieras, sin hacer un commit.

YAML
on:
  workflow_dispatch:  # Aparece botón en Actions

Programado — cron

Se activa automáticamente según un horario. Usa sintaxis cron (minuto, hora, día del mes, mes, día de la semana).

YAML
on:
  schedule:
    - cron: '0 8 * * 1-5'  # Lunes a viernes 8am UTC
04

Jobs y Steps explicados

runs-on:

Sistema operativo del servidor de GitHub. ubuntu-latest es gratuito y el más común.

ubuntu-latest macos-latest windows-latest

uses:

Usa una acción preconstruida del Marketplace. Evita tener que escribir scripts largos.

checkout@v4 setup-node@v4

run:

Ejecuta comandos de terminal directamente. Usa | para múltiples líneas.

npm install npm test

Los steps más comunes explicados:

1
Descargar el código Siempre primero
YAML
- name: Checkout código
  uses: actions/checkout@v4  # descarga el repo en el servidor
2
Instalar Node.js Node.js
YAML
- name: Setup Node.js
  uses: actions/setup-node@v4
  with:
    node-version: '18'
    cache: 'npm'  # guarda caché para ser más rápido
3
Instalar Flutter Flutter/Dart
YAML
- name: Instalar Flutter
  uses: subosito/flutter-action@v2
  with:
    flutter-version: '3.19.0'
    channel: 'stable'
4
Guardar archivos generados (artefactos)
YAML
- name: Guardar build
  uses: actions/upload-artifact@v4
  with:
    name: mi-build
    path: ./dist          # carpeta a guardar
    retention-days: 7  # disponible 7 días
5
Jobs que dependen de otros
YAML
reporte:
  needs: [ validar, build ]   # espera que ambos terminen
  runs-on: ubuntu-latest
  if: always()                # corre aunque fallen
  steps:
    - name: Mostrar resultados
      run: |
        echo 'Validar: ${{ needs.validar.result }}'
        echo 'Build:   ${{ needs.build.result }}'

Plantillas completas

ci.yml — Node.js 
name: CI - Mi Proyecto Node

# ── TRIGGER ──────────────────────────────────────
on:
  push:
    branches: [ main, development, feature/* ]
  pull_request:
    branches: [ main, development ]

# ── VARIABLE GLOBAL ──────────────────────────────
env:
  NODE_VERSION: '18'

jobs:

  # JOB 1: Validar el backend
  validate-backend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./backend
    steps:
      - name: Checkout código
        uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
      - name: Instalar dependencias
        run: npm install
      - name: Ejecutar pruebas
        run: npm test

  # JOB 2: Build del frontend
  build-frontend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./frontend
    steps:
      - name: Checkout código
        uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
      - name: Instalar dependencias
        run: npm install
      - name: Build de producción
        run: npm run build
      - name: Guardar artefacto
        uses: actions/upload-artifact@v4
        with:
          name: frontend-dist
          path: ./frontend/dist
          retention-days: 7

  # JOB 3: Reporte final
  reporte:
    needs: [ validate-backend, build-frontend ]
    runs-on: ubuntu-latest
    if: always()
    steps:
      - name: Resultado del CI
        run: |
          echo "Backend:  ${{ needs.validate-backend.result }}"
          echo "Frontend: ${{ needs.build-frontend.result }}"
ci.yml — Flutter / Dart 
name: CI - Mi App Flutter

# ── TRIGGER ──────────────────────────────────────
on:
  push:
    branches: [ main, development, feature/* ]
  pull_request:
    branches: [ main, development ]

# ── VARIABLE GLOBAL ──────────────────────────────
env:
  FLUTTER_VERSION: '3.19.0'

jobs:

  # JOB 1: Análisis y pruebas
  analyze-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout código
        uses: actions/checkout@v4
      - name: Instalar Flutter
        uses: subosito/flutter-action@v2
        with:
          flutter-version: ${{ env.FLUTTER_VERSION }}
          channel: 'stable'
      - name: Obtener dependencias
        run: flutter pub get
      - name: Análisis estático
        run: flutter analyze
      - name: Ejecutar pruebas
        run: flutter test

  # JOB 2: Build Android (APK debug)
  build-android:
    needs: [ analyze-and-test ]
    runs-on: ubuntu-latest
    steps:
      - name: Checkout código
        uses: actions/checkout@v4
      - name: Instalar Java
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'
      - name: Instalar Flutter
        uses: subosito/flutter-action@v2
        with:
          flutter-version: ${{ env.FLUTTER_VERSION }}
          channel: 'stable'
      - name: Obtener dependencias
        run: flutter pub get
      - name: Build APK debug
        run: flutter build apk --debug
      - name: Guardar APK
        uses: actions/upload-artifact@v4
        with:
          name: app-debug.apk
          path: build/app/outputs/flutter-apk/app-debug.apk
          retention-days: 7

  # JOB 3: Reporte final
  reporte:
    needs: [ analyze-and-test, build-android ]
    runs-on: ubuntu-latest
    if: always()
    steps:
      - name: Resultado del CI
        run: |
          echo "Pruebas: ${{ needs.analyze-and-test.result }}"
          echo "Build:   ${{ needs.build-android.result }}"
